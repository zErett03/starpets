# StarPets — чек-лист боевого прогона (запуск)

> Проходишь как **покупатель** (через тестовую карточку ggsel) + как **оператор** (панель `/admin`).
> Легенда статусов заказа: `pending` (в обработке) · `dispatched` (трейд создан, идёт доставка) · `done`/`finalized` (доставлено, оплата высвобождена) · `failed` (не доставлено) · `needs_attention` (нужна ручная проверка).

## 0. Подготовка (до тестов)

- [ ] Задеплоено; в логах старта видно `alembic upgrade head` без ошибок (миграция **0011** = колонка `dispatched_at`).
- [ ] `/admin` открывается, просит логин/пароль (Basic Auth). Пароль `ADMIN_PASSWORD` применён через **Promote → Shared**.
- [ ] Секреты проротированы (`starpets_secret`, `Api_Key`/`account_id`, `ADMIN_PASSWORD`) — они светились в переписке.
- [ ] Тестовая карточка создана и **активна** на ggsel, привязана к offer в нашей БД (`starpets_product_id` заполнен, есть сток). Проверь `/order-info` позже по факту заказа.
- [ ] `/system-status` отвечает: баланс токена StarPets > стоимости теста, счётчики заказов видны.
- [ ] (опционально, чтобы не ждать по 10 мин в тестах таймера) временно уменьшить `_DELIVERY_TIMEOUT` в `app/workers/monitor_delivery.py` до `minutes=2`, задеплоить, после тестов вернуть **10**.

## 1. Happy path — успешная доставка

- [ ] На карточке ggsel ввод **правильного** Roblox-ника → precheck проходит (qty=1, ник принят).
- [ ] Оплата → редирект на `/delivery` → сначала **спиннер** «Заказ обрабатывается» (авто-рефреш 5с).
- [ ] Через несколько секунд появляется **имя бота**, таймер **10:00** и идёт вниз.
- [ ] **Ник бота кликабелен** → открывается в **новой вкладке** правильный профиль бота на roblox.com.
- [ ] В шаге 2 ссылка **«профиль бота»** тоже открывает профиль. В инструкции написано **Join** (НЕ «телепортируйтесь»).
- [ ] Добавляешь бота в друзья → ждёшь ~1 мин → бот принимает заявку.
- [ ] Профиль бота → **Join** → загрузка в Adopt Me на сервере бота → принимаешь входящий трейд → **предмет получен**.
- [ ] Оператор: заказ виден в `/admin`, `SP статус` доходит до **5 (IN_PROGRESS)**; «История доставки» показывает прогресс `0→2→…→5` + сводку-пруф.
- [ ] **Авто-закрытие**: дождись истечения таймера — монитор пересоздаёт трейд, ловит `130 NOT_FOUND` → заказ авто-переходит в **done**, ставится `MARK_DELIVERED` (оплата на ggsel высвобождается). В логах: `create_trade 130 NOT_FOUND: предмет ушёл — ДОСТАВЛЕНО`.
- [ ] Альтернатива: оператор закрывает вручную кнопкой **«Отправить на ggsel»** до истечения таймера — заказ `done`, оплата высвобождена.
- [ ] Повторный заход на `/delivery` после закрытия показывает **«Предмет доставлен!»**.

## 2. Неверный логин покупателя

### 2a. Указан валидный, но НЕ тот аккаунт (главный кейс)
- [ ] Оформляешь заказ на «чужой»/неправильный существующий ник → доходит до `dispatched`, бот появляется, но принять трейд ты не можешь (это не твой аккаунт).
- [ ] Оператор в `/admin`, строка заказа → блок **«Новый логин»**: ввести **правильный** ник, выбрать причину **«Неверно указан логин»**, нажать **Новый логин** → подтвердить.
- [ ] Ожидаемо: старый трейд **отменён** (`cancel_trade`), создан **новый** трейд на правильный ник, обновились `Trade ID`/`Бот`, статус снова `dispatched`, во flash — результат цепочки.
- [ ] Покупатель на `/delivery` (авто-рефреш 20с) видит **нового бота** и **свежий таймер** → проходит доставку.

### 2b. Несуществующий / битый логин
- [ ] Оформляешь заказ на несуществующий ник.
- [ ] Наблюдаешь: заказ **не** уходит в `dispatched` (create_trade падает), покупатель видит спиннер. В логах — код ошибки создания трейда (напр. `140 EXTERNAL_BUYER_NOT_FOUND` / иной).
- [ ] Исправление оператором: в строке заказа сохранить правильный ник (поле «Roblox ник» ✓), затем `GET /trigger-deliver?order_id=<ID>` (или кнопка). Проверить, что доставка запускается.

## 3. Таймер истёк — покупатель не пришёл

- [ ] Оплачиваешь, получаешь бота, **ничего не делаешь**.
- [ ] По истечении таймера монитор **авто-пересоздаёт** трейд (retry 1): новый бот, таймер сброшен, `trade_retry_count=1`. В логах: `timer expired (retry 1/2)`.
- [ ] Ещё один интервал → retry 2 (`trade_retry_count=2`).
- [ ] Следующее истечение → заказ переходит в **needs_attention** (лог `max auto-retries reached`). Покупатель больше не получает нового бота — дальше решает оператор.
- [ ] Оператор видит `needs_attention` в фильтрах `/admin`, разбирается: «Новый трейд» / «Новый логин» / возврат.

---

## 4. Товар закончился / цена выросла

- [ ] **Нет стока** (`no_items_available`): если на момент выдачи предмета нет — заказ `failed`, на `/delivery` — «Не удалось доставить предмет» (текст без ложного «деньги вернутся автоматически»). Возврат оформляешь на ggsel вручную.
- [ ] **Цена выросла выше оффера** (`price_too_high`, код 330): заказ `failed`, причина видна в `/admin`. (Спровоцировать трудно — достаточно убедиться, что при появлении в логах отрабатывает корректно.)

---

## 5. Отмена / возврат

- [ ] Отмена активного трейда до истечения таймера выполняется через блок **«Новый логин»** (он делает `cancel_trade`) — проверь варианты причины в выпадающем списке, включая **«Другое»** со своим текстом.
- [ ] Чистый возврат без пересоздания: оператор ставит заказу статус **failed** (селект статуса) и оформляет возврат на ggsel вручную.
- [ ] Проверь код `wrong_account_specified` (значение «Неверно указан логин») и `other` (свой текст 1–144 симв.) — оба принимаются StarPets без ошибки (следи за логом `[cancel_trade] FAILED ...`, если контракт обёртки отличается).

## 6. Пограничные / технические

- [ ] **Дубликат вебхука/повторная оплата** — второй `DELIVER` не создаёт вторую покупку (идемпотентность по `id_i` / переиспользование `purchase_id`).
- [ ] **Многократный рефреш `/delivery`** — в логах нет 400-спама `friendship` после того, как бот принял заявку (гейт по `starpets_status`). Таймер **не сбрасывается** при рефреше (серверный якорь `dispatched_at`).
- [ ] **Закрытая вкладка / телефон** — даже без открытой страницы монитор шлёт friendship и ведёт таймер (серверная логика). Доставка проходит.
- [ ] **«Новый трейд» на уже доставленный заказ** → `130` → авто-`done` (кнопка «Новый трейд» в `/admin`).
- [ ] **«Новый трейд», когда предмет залочен в активном трейде** → `210` → `needs_attention` (не ломает живой обмен).
- [ ] **История доставки** наполняется на новых заказах (аудит `trade_events`), сводка копируется кнопкой 📋.

## Полезные эндпоинты для дебага
- `GET /system-status` — счётчики заказов/офферов, баланс.
- `GET /order-info?order_id=<ID>` — состояние заказа.
- `GET /trigger-deliver?order_id=<ID>` — перезапуск полной выдачи (сбрасывает таймер и счётчик ретраев).
- `GET /fix-order-username?order_id=<ID>&username=<НИК>` — сменить ник (+ авто-DELIVER при необходимости).
- `/admin` — статус, «Новый трейд», «Новый логин», «Отправить на ggsel», история.
- Логи Railway — ключевые метки: `[Deliver]`, `[MonitorDelivery]`, `[redeliver]`, `[cancel_trade]`, `[roblox]`.

## Матрица «статус → действие оператора»
| Статус | Что значит | Действие |
|---|---|---|
| pending | в обработке / ждёт выдачи | подождать; если завис — `/trigger-deliver` |
| dispatched | трейд создан, идёт доставка | ничего; таймер сам пересоздаст/закроет |
| done / finalized | доставлено, оплата высвобождена | — |
| failed | не доставлено (нет стока/цена/битый ник) | исправить ник + `/trigger-deliver` или возврат на ggsel |
| needs_attention | таймер исчерпал ретраи / 210 / ошибка | разобрать вручную: «Новый трейд»/«Новый логин»/возврат |
